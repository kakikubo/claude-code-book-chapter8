# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**TaskCLI** - Gitと一体化した開発者向けタスク管理CLIツール

### プロダクトコンセプト
- **ターミナル完結**: GUIに一切切り替えることなく、タスク管理の全操作をターミナル上で完結させる
- **Git一体化**: タスクとGitブランチを1対1で紐付け、ブランチ作成・コミット・PRまでの開発フローを自動支援する
- **ゼロ摩擦**: 1分で使い始められるシンプルさと、開発フローに溶け込む自動化で認知負荷を最小化する

### プロダクトビジョン
開発者がコードを書くことに集中できる環境を作る。
タスク管理のためにGUIツールを開くたびに途切れる集中力と、手作業によるタスク・Git・PRの状態管理という二重管理の負担を解消する。
TaskCLIは開発者のターミナルに溶け込み、タスク開始からブランチ作成・コミット・PR作成・タスク完了までの一連のフローを自動化することで、開発者が「今何をすべきか」と「次に何をすべきか」に集中できる状態を作り出す。

### 目的
- ターミナルとGUIの往復によるコンテキストスイッチを排除する
- タスクとGitブランチを自動連携し、作業中のタスクを常に明確にする
- タスク管理の手作業（ステータス更新、PR作成など）を自動化する
- 小規模チームがJiraやAsanaなどの重厚なツールなしにタスク状況を把握できるようにする

### プライマリーペルソナ: 鈴木健一（29歳、フリーランスエンジニア）
- フリーランスで3〜5プロジェクトを並行して担当
- ターミナル中心の開発環境（Vim / VS Code + 統合ターミナル）
- GitHub Issuesを使っているが、ローカル作業との連携ができていない
- **課題**: 複数プロジェクトのタスクが頭の中だけで管理され、「次に何をすべきか」が不明確になる
- **期待する解決策**: ターミナルで完結し、Gitブランチと自動連携するシンプルで速いタスク管理
- **1日のワークフロー**: `task list` で今日のタスクを確認 → `task start <id>` でブランチ自動作成 → 実装 → `task done <id>` で完了

- JiraやAsanaは機能過多・コスト過多で小規模チームには不向きと感じている
- **課題**: 誰が何をやっているかGUIを開かないと分からない
- **期待する解決策**: CLIでさっとチーム全体のタスク状況を確認でき、GitHub連携で自動更新される軽量ツール

## 機能要件

### コア機能（MVP / P0）

#### 1. タスクの基本CRUD操作

**ユーザーストーリー**:
開発者として、ターミナルを離れずにタスクを管理するために、タスクの追加・表示・完了・削除ができるコマンドが欲しい

**受け入れ条件**:
- [ ] `task add "<タイトル>"` でタスクを追加でき、UUID v4の先頭8文字をユーザー向け表示IDとして採番される（例: `a1b2c3d4`）。ユーザーはこの8文字IDでタスクを指定する
- [ ] `task list` で全タスクをID・ステータス・タイトル・紐付きブランチ付きで一覧表示できる
- [ ] `task show <id>` でタスクの詳細情報（タイトル・ステータス・作成日時・ブランチ）を表示できる
- [ ] `task done <id>` でタスクのステータスを `completed` に変更できる
- [ ] `task delete <id>` でタスクを削除でき、削除前に確認プロンプトが表示される
- [ ] 全コマンドの実行時間が100ms以内（通常のPC環境）

**優先度**: P0（必須）

#### 2. タスクのステータス管理

**ユーザーストーリー**:
開発者として、作業の進捗を追跡するために、タスクに `open` / `in_progress` / `completed` / `archived` のステータスを設定できる機能が欲しい

**受け入れ条件**:
- [ ] タスク作成時のデフォルトステータスは `open`
- [ ] `task start <id>` で `in_progress` に変更される
- [ ] `task done <id>` で `completed` に変更される
- [ ] `task archive <id>` で `archived` に変更される
- [ ] `task list` でステータスをカラーで視覚的に区別できる

**優先度**: P0（必須）

#### 3. Gitブランチとの自動連携

**ユーザーストーリー**:
開発者として、タスクとコードの作業を1対1で管理するために、タスク開始時に自動でGitブランチが作成され、切り替わる機能が欲しい

**受け入れ条件**:
- [ ] `task start <id>` 実行時に `feature/task-<id>-<タイトルのslug>` 形式のブランチが自動作成・チェックアウトされる
- [ ] ブランチ名はASCII文字に変換（日本語タイトルはローマ字/英語slugに変換）
- [ ] 既に同名ブランチが存在する場合は既存ブランチにチェックアウトする
- [ ] `task list` にブランチ名を表示する
- [ ] Gitリポジトリが存在しない場合は警告を表示し、ブランチ作成をスキップしてタスクのみ更新する

**優先度**: P0（必須）

#### 4. シンプルなタスク一覧表示

**ユーザーストーリー**:
開発者として、現在の作業状況を瞬時に把握するために、タスクの状態をテーブル形式で一覧表示できる機能が欲しい

**受け入れ条件**:
- [ ] `task list` でID・ステータス・タイトル・紐付きブランチを表示する
- [ ] デフォルトは `archived` 以外の全タスクを表示する
- [ ] ステータスごとにカラーを使い視覚的に区別する（`in_progress` は黄色、`completed` は緑など）
- [ ] 1000件のタスクでも1秒以内に表示される

**優先度**: P0（必須）

---

### 重要機能（P1）

#### 5. タスクの絞り込み・検索

**ユーザーストーリー**:
開発者として、多数のタスクの中から目的のものを素早く見つけるために、ステータスやキーワードで絞り込める機能が欲しい

**受け入れ条件**:
- [ ] `task list --status <status>` でステータス別に絞り込める
- [ ] `task search "<キーワード>"` でタイトルを部分一致検索できる
- [ ] 絞り込み結果が0件の場合、適切なメッセージを表示する

**優先度**: P1（重要）

#### 6. 優先度と期限の管理

**ユーザーストーリー**:
開発者として、重要なタスクを見逃さないために、優先度（high/medium/low）と期限日をタスクに設定できる機能が欲しい

**受け入れ条件**:
- [ ] `task add "<タイトル>" --priority high --due YYYY-MM-DD` で優先度・期限を設定できる
- [ ] `task list --sort priority` で優先度順に並び替えて表示できる
- [ ] 期限が当日・翌日のタスクを `task list` 表示時に警告表示する

**優先度**: P1（重要）

#### 7. GitHub Issuesとの同期

**ユーザーストーリー**:
開発者として、既存のGitHub Issuesとローカルのタスクを統一管理するために、GitHub Issuesからタスクをインポートし、同期できる機能が欲しい

**受け入れ条件**:
- [ ] `task import --github` でGitHub Issuesをタスクとしてインポートできる
- [ ] `task sync` でローカルタスクとGitHub Issuesの状態を双方向同期できる
- [ ] Personal Access Tokenで認証し、設定は `~/.taskcli/config.json` に保存される
- [ ] GitHub APIのレート制限に達した場合、適切なエラーメッセージを表示する

**優先度**: P1（重要）

#### 8. タスク完了時の自動処理

**ユーザーストーリー**:
開発者として、完了後の手動作業を減らすために、タスク完了時にGitHub PR作成などを自動実行できる機能が欲しい

**受け入れ条件**:
- [ ] `task done <id> --create-pr` でGitHub PRを自動作成できる
- [ ] PRタイトルにはタスクタイトルが、本文にはタスクIDが含まれる
- [ ] `task done <id>` 単体ではPR作成をスキップし、タスクステータスのみ更新する
- [ ] `task done` はブランチのマージ・リモートへのプッシュを自動実行しない（誤マージによるコード破壊を防ぐため、これらは手動操作とする）

**優先度**: P1（重要）

#### 9. GitHub連携の設定コマンド

**ユーザーストーリー**:
開発者として、GitHub連携を安全に設定するために、Personal Access Tokenをターミナルから登録・確認できるコマンドが欲しい

**受け入れ条件**:
- [ ] `task config --github-token <token>` でPersonal Access Tokenを `~/.taskcli/config.json` に保存できる（パーミッション600を自動設定）
- [ ] `task config --github-owner <owner> --github-repo <repo>` でリポジトリを設定できる
- [ ] `task config --show` で現在の設定を確認できる（トークンは先頭4文字以外をマスクして表示）
- [ ] トークン未設定の状態で `task import --github` を実行した場合、設定コマンドの使い方を案内するエラーメッセージを表示する

**優先度**: P1（重要）

### 将来的な機能（Post-MVP / P2）

#### 9. チーム機能
チームメンバーのタスク一覧表示（`task list --team`）、担当者へのアサイン機能。Git/GitHubを通じたチーム状況の可視化。

**優先度**: P2（できれば）

#### 10. 作業時間の記録
`task start` / `task done` で作業時間を自動計測・記録し、`task show <id>` で確認できる。

**優先度**: P2（できれば）

#### 11. カレンダー表示
`task calendar --week` / `task calendar --month` で期限ベースのカレンダービューを表示。

**優先度**: P2（できれば）

#### 12. タスクテンプレート
バグ修正、機能追加などの定型タスクをテンプレート化し、`task add --template bug-fix` で利用。

**優先度**: P2（できれば）

#### 13. タスクの差し戻し（reopen）
`task reopen <id>` で `in_progress` または `completed` 状態のタスクを `open` に戻せる。レビュー差し戻しなど、作業が再開が必要になった場合に使用する。

**優先度**: P2（できれば）

### CLIインターフェース

```bash
# タスクの基本操作（P0）
task add "ユーザー認証機能の実装"
task add "緊急バグ修正" --priority high --due 2025-01-20
task list
task list --status in_progress
task show a1b2c3d4     # UUID先頭8文字でタスクを指定
task start a1b2c3d4    # ステータスをin_progressに変更し、Gitブランチを自動作成
task done a1b2c3d4     # ステータスをcompletedに変更
task done a1b2c3d4 --create-pr  # タスク完了 + GitHub PR自動作成
task delete a1b2c3d4
task archive a1b2c3d4

# 絞り込み・検索（P1）
task list --status in_progress
task list --sort priority
task search "認証"

# GitHub連携設定（P1）
task config --github-token <token>
task config --github-owner <owner> --github-repo <repo>
task config --show

# GitHub連携（P1）
task import --github
task sync

# ヘルプ
task --help
task <subcommand> --help
```

## 非機能要件

### パフォーマンス
- コマンド実行時間: 全コマンドで100ms以内（平均的なPC環境、ネットワーク不要の操作）
- タスク一覧表示: 1000件まで1秒以内
- GitHub API呼び出しを伴うコマンドは5秒以内、タイムアウト時はエラーメッセージを表示

### ユーザビリティ
- `npm install -g taskcli` の1コマンドでインストールでき、追加設定なしに `task add` / `task list` が動作する（Git連携機能を除く）
- 新規ユーザーが `task --help` なしで基本操作（add/list/start/done）を5分以内に習得できる
- エラーメッセージは原因と推奨する対処法を含む
- 危険な操作（delete）は確認プロンプトを必須とする
- `task <subcommand> --help` で各コマンドの詳細ヘルプを表示できる

### 信頼性
- タスクデータの損失ゼロ（操作前に自動バックアップ `.task/tasks.json.bak` を作成）
- Gitエラー発生時はタスクのステータス変更をロールバックし、データ不整合を防ぐ
- Gitリポジトリが存在しない環境でも基本的なタスク操作（Git連携機能を除く）が動作する

### セキュリティ
- GitHub Personal Access Tokenは `~/.taskcli/config.json` に保存し、パーミッションを600に設定する
- トークンはログや標準出力に一切表示しない
- コマンドインジェクション対策として、タスクタイトルをシェルコマンドに直接渡さない

### スケーラビリティ
- タスク数1万件まで正常動作（パフォーマンス要件を維持）
- MVPのデータ形式はJSON（`.task/tasks.json`）、将来的にSQLiteへの移行パスを設計上確保する

### クロスプラットフォーム対応
- macOS、Linux、Windows（Git Bash / WSL2）で動作する
- Node.js v18以上が必要（インストール手順をREADMEに明記）

## スコープ外

明示的にスコープ外とする項目:
- GUIアプリケーション・Webインターフェース（CLIに特化する）
- GitLabやBitbucketとの連携（MVPはGitHub専用）
- 専用クラウドサービスやクラウド同期（Git経由の共有のみサポート）
- チームメンバー間のリアルタイム同期（非同期のGit/GitHub連携のみ）
- プラグインシステム（将来検討）
- Slack・Discord等の外部通知連携（将来検討）
- Windows ネイティブ（PowerShell / cmd.exe）のサポート（Git Bash / WSL2を推奨）
